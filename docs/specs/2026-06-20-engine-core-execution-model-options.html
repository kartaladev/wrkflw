Context & what we're deciding

wrkflw is a library-first Go workflow engine (Go 1.25). The deliverable is an importable module; there is no daemon we own. The load-bearing constraint that drives this decision:

The engine core — the token state machine that moves a process instance across BPMN nodes — must stay pure of transport, storage vendor, and event-bus specifics. It depends on interfaces (or pure data) only, and must be consumable with no Postgres / watermill / gRPC imported at all. Persistence is the source of truth; events publish via a transactional outbox; timers/SLAs run on gocron; time is controlled via jonboulle/clockwork (which gocron v2 also accepts).

The question on the table: how does the core actually advance an instance? Three candidate shapes are below. This choice shapes testability, crash-recovery, the outbox, the rollback/compensation feature, and how ergonomic the embedded API feels. BPMN scope for core #1 is already decided as broad coverage (all gateways, intermediate/boundary events, sub-processes) — the execution model must comfortably carry that.

Shared concepts (true for all three options)

Regardless of which model we pick, the core works in terms of tokens and a process definition. A token sits on a node and carries instance variables; gateways read those variables (via expr-lang/expr) to decide routing. The differences below are purely about who performs side effects (calling a service action, scheduling a timer, emitting an event, parking on a human task) and how a result re-enters the engine.

// Vocabulary shared by the examples on this page.
type ProcessDefinition struct { /* nodes, sequence flows, gateways, events */ }
type InstanceState     struct { /* live tokens + process variables (what we persist) */ }

type ServiceAction interface {        // the catalog interface, referenced by name in definitions
    Do(ctx context.Context, in map[string]any) (out map[string]any, err error)
}

Option A — Pure stepper + commands as data recommended

The core is a single deterministic, non-blocking function. Given the current state and one trigger ("something happened"), it returns the new state and a list of commands ("things the outside world should do"). Commands are plain data — not callbacks — so the core performs zero I/O and spawns zero goroutines. A thin runtime (a later sub-project, not part of the core) executes the commands and feeds each result back as the next trigger.

// The ENTIRE I/O-free surface of the core.
func Step(def *ProcessDefinition, st InstanceState, stim Trigger) (StepResult, error)

type StepResult struct {
    State   InstanceState   // new token/variable state → runtime persists this (source of truth)
    Commands []Command        // pure-data descriptions of side effects to perform
}

// Trigger = "something happened the engine must react to" (a sealed set).
type Trigger interface{ isTrigger() }
type StartInstance   struct{ Vars map[string]any }
type ActionCompleted struct{ CommandID string; Output map[string]any } result coming back
type ActionFailed    struct{ CommandID string; Err string }              result coming back
type TimerFired      struct{ CommandID string }                          result coming back
type HumanCompleted  struct{ TaskToken string; Decision map[string]any } result coming back
type SignalReceived  struct{ Name string; Payload map[string]any }

// Command = "something the engine wants done" (a sealed set, no funcs inside).
type Command interface{ isCommand() }
type InvokeAction  struct{ CommandID string; ActionName string; Input map[string]any }
type ScheduleTimer struct{ CommandID string; Token string; FireAt time.Time } from clockwork
type EmitEvent     struct{ Topic string; Payload map[string]any }            → outbox
type AwaitHuman    struct{ TaskToken string; Assignee AuthzSpec; SLA *time.Time }
type Compensate    struct{ FromNode string }
Strengths
  • Maximally pure & testable — a step is just data-in/data-out; assert on returned commands, no mocks.
  • Wait states are free — AwaitHuman+ScheduleTimer park the instance in the DB; nothing held in memory.
  • Outbox is natural — runtime persists State and writes EmitEvent in one transaction.
  • Crash recovery / replay — deterministic on (state,trigger); re-deliver after crash, replay a trigger log to reconstruct any instance.
  • Compensation & SLA map to Compensate / TimerFired commands directly — serves the rollback + 3-working-day examples.
Costs
  • Indirection — consumers don't call action.Do() themselves; they run (or we ship) the loop.
  • Command/trigger taxonomy must be designed up front and kept closed.
  • Correlation bookkeeping (CommandID ↔ result) lives in the core's parked-token state.
  • Slightly more ceremony for a trivial "just run this 3-node flow inline" embed.

Option B — Active engine owns execution

The core holds injected interfaces (ActionInvoker, Scheduler, EventSink, Clock) and drives execution itself: it calls service actions inline, schedules timers, spawns/awaits goroutines, and writes events as it walks the graph.

type Engine struct {
    actions   ActionInvoker
    scheduler Scheduler     // wraps gocron
    events    EventSink     // wraps outbox/watermill
    clock     clockwork.Clock
}

func (e *Engine) Advance(ctx context.Context, inst *Instance) error {
    // walks nodes, calls e.actions.Invoke(...), e.scheduler.At(...), e.events.Emit(...)
    // blocks / parks / resumes internally.
}
Strengths
  • Most ergonomic for a simple embed — call Advance and it just runs.
  • Fewer moving parts to wire for the happy path; no separate runtime loop to ship.
  • Familiar imperative control flow for contributors.
Costs
  • Couples the core to concurrency & lifecycle — goroutines/locks live inside the "pure" core.
  • Outbox is awkward — event emission is interleaved with logic, not bundled with the state write in one tx.
  • Replay/determinism is hard — behavior depends on injected I/O timing, not just data.
  • Tests need mocks for every port; timer/SLA tests get fiddlier.
  • Risks violating the CLAUDE.md rule that the core stays free of transport/storage/bus specifics.

Option C — Hybrid: pure transitions + injected ports

A middle path. The core computes transitions purely, but during a single step it calls out synchronously through a few narrow injected ports (e.g. ActionInvoker, Clock, EventSink) rather than returning a command list. No goroutines inside the core, but it does touch I/O ports inline.

type Ports struct {
    Invoke func(ctx context.Context, name string, in map[string]any) (map[string]any, error)
    Emit   func(topic string, payload map[string]any)
    Clock  clockwork.Clock
}
func Step(def *ProcessDefinition, st InstanceState, stim Trigger, p Ports) (InstanceState, error)
Strengths
  • No concurrency inside the core (unlike B); transitions still mostly testable.
  • More direct than A — results return inline from the port call, no CommandID correlation.
  • Less up-front taxonomy than A's full command set.
Costs
  • Still does I/O mid-step — outbox-in-one-tx and clean replay are weaker than A.
  • Determinism leaks through the ports; tests still need port fakes.
  • Park/resume for long human waits doesn't fit a synchronous port call as cleanly as A's AwaitHuman.
  • "Middle ground" risks the worst of both as features grow.

How a result flows back (Option A in motion)

The key mechanism: every command that yields a result carries an CommandID, and the matching result-trigger echoes that same CommandID back. That correlation is the whole trick. The runtime is the only thing that blocks or does I/O:

Step(state, stim)→ persist(State)→ run Commands→ result as next Trigger
(same CommandID)↺ Step(state', stim') 
// The runtime — a LATER sub-project. The core does NOT contain this loop.
for {
    res, err := core.Step(def, state, stim)        // 1. pure decision (no I/O)
    persist(res.State)                              // 2. DB is source of truth
    for _, e := range res.Commands {                 // 3. perform side effects
        switch e := e.(type) {
        case InvokeAction:
            out, err := catalog.Resolve(e.ActionName).Do(ctx, e.Input)
            // result becomes the NEXT trigger, tagged with the SAME CommandID:
            enqueue(ActionCompleted{CommandID: e.CommandID, Output: out}) // or ActionFailed
        case ScheduleTimer:
            scheduler.At(e.FireAt, func(){ enqueue(TimerFired{CommandID: e.CommandID}) })
        case AwaitHuman:
            // no immediate result — instance PARKS in DB. A future REST/gRPC
            // "complete task" call injects HumanCompleted{TaskToken:...} later.
        case EmitEvent:
            outbox.Write(tx, e)                      // same tx as persist(); relayed by watermill adapter
        }
    }
    stim = dequeue() // blocks until next trigger: a result, a signal, or a human action
}

So a result is never "returned" synchronously from Step — it arrives on the next call as ActionCompleted / TimerFired / HumanCompleted carrying the CommandID. The core finds the token parked on that CommandID, merges the output into instance variables, and advances. Waiting 3 working days for a human is simply: emit AwaitHuman + ScheduleTimer(SLA) + optional in-wait ScheduleTimer reminders, then return. The instance lives entirely in the DB until a trigger wakes it.

Where clockwork lands

Time is never read with the wall clock inside the core. jonboulle/clockwork gives a single injectable Clock, and gocron v2 accepts the same interface — so under test one clockwork.NewFakeClock() advances both engine logic and the real scheduler in lockstep.

  • Option A: the core emits ScheduleTimer{FireAt} (or a duration); FireAt is resolved against the injected clock at the boundary. The core itself stays time-free — easiest to test.
  • Options B / C: the clock is an injected port the core reads inline; still fake-able, but the timing lives inside the core's control flow.
Likely worth an ADR: "Use clockwork.Clock as the engine's sole time source; share one instance with gocron." Sits alongside the existing gocron v2.21.2 hard pin.

Recommendation

Option A — pure stepper + commands as data. It is the only option that keeps the core genuinely free of transport/storage/bus (the CLAUDE.md hard rule), and it makes four separate requirements fall out almost for free: the transactional outbox, retry/resilience, compensation/rollback, and long human/SLA waits. The cost — a runtime loop instead of an inline Advance() — is paid once, in a later sub-project, and we can ship a default loop so simple embeds stay one-liner-ish.

If the team prioritizes day-one embed ergonomics over replay/outbox cleanliness, Option C is the fallback. Option B is discouraged: it bakes concurrency and I/O into the part of the system we most need to keep pure.

Open questions for the team

  • Do we accept shipping a default runtime loop in pkg/ so Option A embeds stay ergonomic, or is an inline Advance() a hard requirement?
  • Should the trigger log be a first-class, persisted artifact (full event-sourcing) or just a transient delivery queue? Affects replay/debug depth.
  • For broad BPMN coverage in core #1, do boundary events & sub-processes change the command taxonomy (e.g. a SpawnSubInstance command)? Worth confirming the taxonomy is closed enough.
  • Is per-command idempotency keying (CommandID) sufficient for at-least-once action delivery, or do we also need action-level idempotency tokens?
  • Granularity of a "step": one trigger → run until the next wait (macro-step), or one node at a time (micro-step)? Affects persistence frequency and observability granularity.
